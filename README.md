# Criminal Justice Hub – Service Design

This repository supports the **Criminal Justice Hub Service Design Team**.

We use this repository to manage our sprint board, backlog, service design work, content tasks, user research, technical tasks, team/admin work, and ways of working.

The repository works alongside our GitHub Project board.

## How our work is structured

We organise work using this structure:

```text
Theme → Initiative → Epic → Story / Task
```

| Level | Meaning |
|---|---|
| Theme | A broad strategic area of the service |
| Initiative | A focused area of improvement within a theme |
| Epic | A larger body of work made up of smaller tasks or stories |
| Story / Task | A specific piece of work that can be progressed by the team |

## How to create a new item of work

1. Go to the **Issues** tab.
2. Select **New issue**.
3. Choose the most suitable issue template.
4. Complete the required fields.
5. Add enough detail for the team to understand the work.
6. Link the issue to the relevant theme, initiative or epic where possible.
7. Add the issue to the Service Design Project board if it is not added automatically.

## Definition of Ready

A task should only be pulled into a sprint when it is ready.

A task is considered ready when:

- the title and description are clear and concise;
- the goal or purpose is understood;
- acceptance criteria are defined and testable;
- relevant designs, links or documents are attached;
- dependencies are identified;
- the task is broken down appropriately;
- the scope is well-defined;
- it has been prioritised using MoSCoW;
- it is aligned with team goals, epics or initiatives;
- the team has reviewed and agreed it is ready;
- the task is assignable.

## Definition of Done

A task is considered done when:

- all acceptance criteria are met;
- it has been peer-reviewed or tested;
- supporting documentation has been updated;
- the relevant workflow or release step has been completed;
- the issue has been updated and moved to the correct status.

Different types of work may have additional checks, including technical testing, legal review, accessibility checks, content review or research synthesis.

## MoSCoW prioritisation

We use MoSCoW to prioritise work.

| Priority | Meaning |
|---|---|
| Must Have | Critical to success, launch, compliance or core functionality |
| Should Have | High value, but can be deferred without major impact |
| Could Have | Useful or desirable, but not essential |
| Won’t Have for now | Not planned for the current goal, sprint or release |

## T-shirt sizing

We use T-shirt sizing to estimate rough effort and complexity.

| Size | Approximate effort |
|---|---|
| XS | Less than 1 hour |
| S | 1–5 hours |
| M | 10–15 hours / 1–2 days |
| L | Up to 1 week |
| XL | More than 1 week |

A standard content page is usually treated as **Medium**.

Anything sized **XL** should normally be broken down into smaller stories or tasks before being accepted into a sprint.

## Team rule

If work is not in GitHub, it is not in the sprint.

If work is raised in Teams, meetings or informal discussion, the Product Manager or Delivery Manager should help turn it into a GitHub issue so it can be prioritised and tracked properly.
